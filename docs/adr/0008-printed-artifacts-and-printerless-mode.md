### Printed artifacts catalog and printerless-mode handling

Vajra prints exactly two kinds of artifact at the counter — a **Sale Invoice** and a **Credit Voucher** — and only at the moments those entities are finished. Every Sale produces a Sale Invoice (business copy always, customer copy default-on with opt-out). A Credit Sale additionally produces a Credit Voucher that the customer signs and the shopkeeper keeps. No other transaction type prints: Purchases, Receipts, Payments, Expenses, Income, and Stock Transfers commit silently and surface only through the live UI and the End of Day Report. The End of Day Report itself is an XLSX file (per ADR-0006), not a counter slip.

A finished transaction is committed *after* the printer driver acknowledges the print job, not before — so a driver-side failure leaves no orphan row, and a driver-side success leaves a recoverable record even if paper jams downstream. The Sale Invoice is reprintable from the cashier UI only until the next transaction starts; the Credit Voucher is never reprintable because the customer's signature is what makes it valid. When a single transaction encounters a transient print failure, Vajra prompts *"printer not responding — continue without slip?"* and proceeds for that one transaction only. For sustained printer-unavailability, the shopkeeper toggles **Printerless Mode** in settings; Vajra then commits without printing and instead displays the would-be Sale Invoice and Credit Voucher details on screen, including the Sale Number and Voucher Number, so the cashier can write manual copies by hand.

We chose this narrow print surface because the goods-moving moments (Sale, Credit Sale) are the only ones where a third party — the customer — needs paper to walk away with. Payments, Receipts, and Expenses already produce their own paper artifacts in the real world (a supplier's bill, an electricity board's receipt, a customer's note). Printing Vajra-side duplicates of those would clutter the workflow without adding evidence. The two-mode print-failure handling — per-transaction prompt vs settings-level toggle — keeps a temporary jam from accidentally committing the cashier to manual writing forever, while still giving a clean operating mode for when the printer is genuinely out of action for the day.

#### Consequences

- Telugu-first rendering (ADR-0003) and the blank-for-handwriting rule apply to both the Sale Invoice and the Credit Voucher. Layouts must reserve consistent space for Telugu fields whether or not Telugu translations are present.
- Printerless Mode and the per-transaction skip prompt must both display the Vajra-generated Sale Number / Voucher Number, since the cashier may need to handwrite these on a manual copy and the shopkeeper will reconcile by these numbers.
- Adding a new transaction type later (or deciding Purchases should print after all) means updating this ADR — the current catalog is intentionally closed, not a default-on shape.
- The "next transaction starts" boundary for Sale Invoice reprinting must be defined explicitly in code (cart-open? finish-clicked? first-line-added?) — chosen to match the cashier's mental model of "I'm done with the last one."
